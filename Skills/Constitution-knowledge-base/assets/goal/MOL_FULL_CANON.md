#### A. 当前目标与平台定位

##### MOL目标与交互定位

本文档用于锚定最小可运营上线（MOL）的当前目标：构建一个可持续扩展的对话式机器人底座，以“Hub（中枢）+ 可热插拔业务机器人”的方式增长，并以 Telegram 为第一优先入口与默认终端用户容器。本项目采用双主线文档体系：开发线（定义要开发什么与如何交付）+ 运营线（定义 AI 如何运营中枢、如何调度外部业务机器人、如何治理后台与日志）。在 Telegram 内，终端体验是 chat + Web App 的混合面：chat 负责意图确认、引导与回执，Telegram Web App（HTML5）作为用户可交互的 canvas。机器人能力本体以统一后端 API 形式提供；入口渠道（Web/Telegram/WhatsApp/Facebook 等）按能力支持逐步适配；平台必须支持按“渠道 × 机器人/能力集”的启用/禁用开关，禁用时能力在该渠道不可触达（不仅是 UI 隐藏）。`OctuposOS_Runtime_Backend` 必须与代码仓一起作为项目部署单元上线，因为它是 AI 运维与审计判定的权威依据，而不是可选附件。

##### 多业务平台覆盖范围

补充已裁决：本项目同时是公司统一的多业务平台底座（对内运营 + 对外服务），至少覆盖三条业务线并允许扩展更多业务包：蒜蓉酱经营管理（库存/价格/销售/税务/流水等）、招聘平台/ careerbot、签证自动化与上门服务（对话式客服机器人 + 履约工作流）。终端用户以统一账号登录后，可聚合使用公司所有业务（类似“一个账号使用公司全家桶”）；用户侧除 Telegram Web App 外，也必须支持独立的 H5/桌面网站入口用于资料管理、配置与下单等交互。

##### 愿景叠加与可追溯原则

产品演进方式（愿景叠加）：本蓝图允许在同一愿景基线上持续叠加新的业务域与运营约束；每次新增内容视为“在既有平台之上新增可交付能力包”，而不是推翻底座重做。因此文档与产物链路必须长期可追溯，避免后期因数据量增大而无法合规导入或回放。

##### 模板复用与实例化机制

平台底座强制设计理念（模板可复用）：管理后台的各业务域能力必须以“可配置模板（template）+ 可实例化项目（project instance）”的方式设计，确保能力可在新项目上复用而不需要重新设计一套 cashflow/税务/报表等基础模块。以蒜蓉酱为首个落地场景：其财务体系与税务计算必须先按菲律宾税务体系固化为一个可配置模板，并支持在未来对另一个项目创建“独立财务实例”（独立数据库/表/命名空间/审计域），仅通过必要配置即可复用同一能力实现。类似地，插件型能力（如 Google Sheets 填表、生成 PDF、表单化录入/导出）必须以模板与可复用功能块形态落地：一次实现可在不同项目实例上启用并生成该项目自己的独立数据与产物。

##### 本轮固化对象与边界

本轮固化的对象与范围是：1 个 Hub（中枢，常驻不下线）与一组可热插拔的“业务包/能力包”（可独立启停、可替换、可扩展）。Hub 同时承担三件事：1）调度 AI 与业务机器人；2）作为你与 Admin Panel 的对话式操作入口；3）记录人工点击与 AI 代理动作的统一日志，回流给 AI 做状态认知。业务包既包含对话式机器人，也包含面向用户与运营的 web 前后端与业务数据治理能力；共同底座是统一身份、权限与审计，以及对话/回执与跨渠道入口治理。此处“固化”指产品边界、对象模型与交互合同，不等价于实现细节承诺。

#### B. 上线顺序与实现分轨

##### 六阶段上线路线

里程碑拆解（L1）在本轮必须显式遵循统一时间线，避免把“底座建设、交互固化、业务扩展、运营放量”混在同一波次。本项目采用双轴同线推进：实施轴（Phase 1-3）约束研发顺序与门禁，业务轴（阶段一至六）约束能力包扩展；两者不是并列路线，而是同一条交付链路的不同切面。当前按窗口推进为：W1（完成 Phase 1 并落地业务阶段一）、W2（完成 Phase 2 并推进业务阶段二与阶段三）、W3（完成 Phase 3 后推进业务阶段四至阶段六）。

阶段一：先完成平台底座（Hub/统一身份/权限与审计/渠道开关/回执与回放/基础可观测），并且前置打通“super admin（唯一）通过 Telegram 与 AI 对话管理后台”的运维闭环：在服务端安装并运行 Codex CLI，由你通过对话驱动服务端 Codex CLI 访问运行时信息，执行配置治理、开关治理、排障与调优。后台对 AI 暴露的可操作面必须以“skills 类结构化能力”形式提供（可门控、可审计、可回放）。强边界：Codex CLI 运维代理只能调用“后台显式提供的管理能力”，不得绕过后台直接改生产代码或直连数据库做未授权写操作；AI 可以程序化触发后台能力，但其副作用必须完全受后台能力/权限/审计约束。

阶段一补充运行架构裁决：章鱼OS在本项目中的执行 AI 为“开发+运维一体”的 `octopusos_ai_operator`，先把原本仅在本机 WSL 交互使用的 Codex 迁移到项目后端线程运行时（`backend_thread_runtime`），并由本项目日志域统一产出运行审计。初期允许以 laptop 作为本地服务器承载该后端；后续迁移到远程服务器时必须保持同一命令协议、同一审计字段和同一回滚语义。WSL 交互环境保留为开发壳层，不作为长期生产运维执行面。

阶段一补充部署裁决：`OctuposOS_Runtime_Backend` 与 `Octopus_CodeBase_Backend` 是同一发布切片的并行产物，必须同版部署、同版回滚、同版审计。任何“仅代码更新不更新 runtime 合同”或“仅文档更新不更新代码映射”的发布都视为不合规发布并阻断放行。

阶段二：交付蒜蓉酱业务“可运营闭环”（客户前端 + 管理后台），覆盖库存/价格/销售/税务/流水管理，并支持 QRCode 引流、文案存储与调用。合规前置：财务/税务/流水等高风险动作必须纳入权限治理与审计闭环，避免后期数据扩张后难以规范导入与追溯。

阶段三：上线公司业务介绍/客服机器人，作为对外咨询与导流入口，并承担多业务默认指引与客服升级枢纽。

阶段四：上线签证自动化能力（对话式机器人 + 自动化流程），至少包含两类机器人：机器人1用于菲律宾旅游签证续签自动化；机器人2用于自动化生产工作签证申请材料（材料收集、抽取、校验、生成与回执）。

阶段五：上线招聘机器人（careerbot），并按需扩展招聘业务的用户侧与运营侧前后端能力（资料/流程/交付与审计）。

阶段六：扩展其它可聚合“AI 应用式服务平台”业务包（非传统 SaaS，只要能聚合给 AI 的能力都纳入聚合），例如二手物品售卖、附近餐厅推荐/点评等。

实施轴（Phase 1-3）在上述业务推进中执行硬门禁：  
Phase 1（管理入口底座）固定交付 Telegram 通道能力、后端命令编排与 payload normalize、开发者 Web UI Admin（人类手动 + AI 代理双模式）；  
Phase 2（Web Codex CLI 交互）要求 Web UI 提供类本地终端能力（发消息、调技能、触发 splash command、查看执行流与回包），且语义对齐本地 Codex CLI；  
Phase 3（Telegram 多机器人桥接）要求多个 Telegram 机器人映射多个 Codex 会话并稳定中转命令回包，所有 Telegram payload 统一归一为同一命令协议。

门禁要求：每阶段必须满足可用性、P95 时延、命令成功率、会话隔离、审计可追踪五类指标并通过内部稳定性验证，未达标不得进入下一阶段；运营放量顺序必须与技术顺序一致，禁止先扩量后补底座。

##### 终极交互目标与入口降级

终极目标（方向性，不在早期阶段承诺交付）是：AI 将网站页面视为 canvas，通过与用户对话持续切换与呈现不同内容，让用户在页面上点击/操作/下单/工作；AI 像导购一样根据用户需求动态组织展示与交互。该能力天然依赖支持 web 端 UI 的入口（例如 Telegram WebApp 与浏览器）；当目标平台不支持 web 承载面时，系统必须明确告知用户“使用支持 web 的入口以解锁完整能力”，并对不支持 web 的入口执行能力降级与清晰回执。

##### 运维与业务实现分轨

已裁决（实现分轨）：平台必须长期保持两条并行但不混写的语义链路。第一条是运营链路：你通过对话驱动服务端 Codex CLI 操作 Admin Panel，AI 代理动作与人工点击动作统一入日志并可追溯；这条链路用于后台治理、配置、门禁、排障与运营调度。第二条是用户链路：终端用户通过对话入口消费业务能力，机器人通过 API + WebApp canvas 提供服务并返回用户回执。两条链路必须拆分 prompt 资产：后台运维 prompt 与前台用户流程 prompt 不能混用、不能共享同一决策语义。

#### C. 试点入口与管理隔离

##### Telegram试点与Chat-Web协同

试点从 Telegram 开始，并将 Telegram 视为主要终端用户容器。用户使用方式固定为“对话驱动 + 页面承载”：chat 负责解释、确认、追问与回执；Telegram Web App 作为 canvas 承载用户必须看到并可交互的页面（历史、设置、登记、订单、审核等）。用户不应被迫离开 Telegram 才能完成关键流程。chat 可利用 Telegram 原生能力（inline buttons、deep links）打开并驱动到特定 Web App 视图，同时保持对话是权威的可审计承载点。

##### 管理后台隔离与访问约束

运营与开发控制必须与终端用户表面严格隔离。提示词（prompt）文本、变量、工作流（workflow）组合、以及健康/状态可见性应通过远端部署的管理后台（Admin Console）管理。平台对内与对外共享一套账号体系：工作人员账号由普通用户账号“被赋权”而来，只有在后台显式开通管理权限后才出现管理入口；未开通前该账号必须表现为纯普通用户，且无法枚举后台资源。后台访问必须启用设备绑定（换设备需重新绑定，否则拒绝登录）。运营动作支持两种入口：人工鼠标点击与 AI 代理执行；两者都必须写入统一审计日志并回流到 AI 上下文，避免“人工改了但 AI 不知道”。终端用户永不直接访问管理后台。

##### Super Admin权限边界

已裁决（后台形态与权限隔离）：所谓“运营后台”本质上是 super admin（唯一 owner）的管理后台；平台只存在一套管理后台入口与页面体系（逻辑隔离，不做早期物理隔离）。其他被赋予权限的账号在同一套后台中只看到其被授权的能力与数据视图，且必须看不到 super admin 才能看到的内容；这些隐藏能力必须不可枚举（不仅仅是隐藏菜单），以降低误配权限与探测风险。对财务/税务/流水等高风险动作，必须启用额外门禁与强审计（例如 step-up challenge/二次确认/短时授权等），并明确支持“由你决定不让 AI 负责”的操作范围（你可在后台手动操作；AI 不得越权）。

#### D. Hub 增长模型与阶段职责

##### Hub增长模型与职责解耦

系统预计会通过“Hub + 可插拔机器人 + 可维护的工作流节点内容”来增长，而不是把能力锁死为刻板的“照标题走（follow-the-headings）”行为。Hub 是注册中心与调度中枢：负责机器人发现与启停、统一路由与导航骨架、身份锚点与访问策略、以及可审计回执与回放。业务机器人是可被替换/禁用的业务包：业务包是可独立交付的能力单元，必须显式拆成数据状态、流程编排、前后端协同、模型裁决链等可验证维度；它们只通过 Hub 的合同与共享基础能力交互，不应形成强耦合链路；单个机器人不可用不应导致整体不可用。以下“第一/第二/第三阶段职责”用于描述能力成熟序列，并与 B 节时间线交叉对齐。目标是在保持意义稳定的前提下，让运行时按需组合这些职责。

##### 第一阶段职责

第一阶段职责是：可靠的入口、Hub 基线与最小运营基线。一旦 Telegram 私聊或群组入口被触发，系统执行握手流程，展示设定预期的欢迎信息，并通过 Hub 提供连接 chat 与 Telegram Web App 的导航骨架（统一的个人面板入口、以及到具体业务机器人的可见路由）。系统提供清晰的人类坐席升级路径，生成最小化回执，让失败与健康状态在语义上可见，并为关键操作留下可追溯痕迹。运营可通过管理后台发布并渐进式灰度提示词（prompt）变更，同时观察健康指标与失败告警。在这一阶段，变量存储、知识检索、以及材料上传明确不在范围内。

##### 第二阶段职责

第二阶段是：受控的上下文存储、身份锚点与可工具化能力。系统将对话与关键抽取事实写入持久存储（database）与短生命周期存储（cache），使变量替换与对话中的受限 tool calls 成为可能。此阶段必须建立“未注册临时 ID → 注册用户”的连续身份锚点，使对话回放、草稿状态与业务产物可以被正确归属、导出与治理。运营可以在管理后台中定义变量占位符、工具边界、以及变量命中率的告警。该阶段包含消息持久化、短期 cache 片段（最近消息与草稿状态）、数据库读写工具、变量命中监控、回放链接、失败回执、以及变量/工具配置。知识引用与模板生成在此阶段仍不在范围内。

##### 第三阶段职责

第三阶段是：知识集成，作为一致回答的机制。知识的权威来源是 `Rise/KnowledgeBase` 下的文档语料及其 Index；回答应当可追溯到具体条目与段落。运营通过管理后台管理知识源、“hot zones”、以及回答一致性面板；文档命中需要被记录。材料上传、模板生成、以及增长/推送行为不属于该阶段职责。

##### 材料治理阶段职责

在获得用户明确同意后，系统可收集并治理材料。通过 chat（以及 Telegram Web App），试点可以接收证件、PDF、照片等文档；抽取字段；并提示用户修正缺失或模糊的值。这也意味着必须可执行的用户权利与可审计的处理方式：查看/导出/更正/删除/撤回，并带完整审计轨迹。该阶段包含：带占位符校验的上传入口、字段抽取、分步修正提示、用户回执、审计日志、以及在 Admin 侧对字段来源与预期用途的解释。模板起草与营销推送不在范围内。

##### 模板生成阶段职责

当字段可用后，系统可以生成基于模板的文档。目标是基于既有 PDF 或表格模板与已收集字段生成申请草稿，并支持在 chat 中预览、下载与提交，同时记录字段溯源。运营需要观察模板命中率与失败归因。该阶段包含：模板到字段映射管理、预览对话触发（带明确的“view/back”动作）、字段来源标注、下载/打印通道、命中/失败指标与回执。该阶段不扩大数据收集范围，也不引入增长推送。

##### 体验增强与合规增长职责

最后才是体验增强与合规增长，并且必须保持边界清晰。在合规与频率控制下，系统可提供 chat 内简化表单、进度/材料提醒、主动通知，以及明确的退订机制。运营通过管理后台配置外联策略、冷却窗口、退订回执，并同时维护外联效果归因与配额控制。该阶段不允许无差别跨渠道扩张，也不得执行未授权的营销。

#### E. 横向约束、用户群体与租户策略

##### 知识冲突裁决与数据最小化

知识处理是贯穿上述阶段的横向硬约束。当知识与提示词（prompt）指令发生冲突时，应以更新更晚的文档版本为权威，且回应必须清晰说明其依据。用户数据与材料仅能在明确同意与合法范围内使用；敏感内容需执行“最小必要”的检测、掩码与告警。

##### 三类用户群体职责

本平台存在三类明确的用户群体。终端用户在不同业务机器人中完成不同任务：客服机器人回答公司业务与引导；签证延期机器人协助准备材料、自检准备度并分步补全；招聘面试机器人在用户授权下对简历/材料进行分析并给出可执行建议与下一步动作。运营者/管理员通过管理后台维护提示词（prompts）、知识与工作流（workflows），并管理入口策略、机器人启停、变量/工具范围、健康监控、以及场景/模板发布；管理权限的开通与设备绑定为后台访问的硬门禁。人类坐席在低置信度回复、敏感话题、或用户明确请求时接管，携带上下文完成案件（case）闭环，并产出会后跟进与总结。

##### 多租户与服务策略差异

多租户（multi-tenant）是平台层面的前提，但不同业务机器人允许采用不同的服务策略。至少存在两类常见策略：面向个人用户（consumer）与面向企业用户（enterprise）。平台提供统一的身份锚点、权限与审计能力，以支持同一机器人在不同策略下的差异化规则；同时也允许某些机器人仅面向其中一类用户。就本轮固化范围而言：客服机器人面向所有终端用户；签证续签机器人与招聘面试机器人需要同时覆盖个人用户与企业用户（并允许在机器人内部定义其特定规则）。此处固化的是“平台必须支持两类策略并可在机器人层表达差异”，而不是具体的计费、合同或实现方式。

##### 产物可见性分级治理

产物与分析结果的可见性必须是“按机器人与按输出类别”治理，而不是平台一次性放开。签证相关的关键抽取字段与缺失项应对用户可见并需要用户确认/更正；招聘面试机器人的部分分析结果可能仅对内部可见（例如用于质量评估、归因或风控），不应默认展示给用户。平台必须支持这种可见性差异，并通过回执与面板在语义上表达“用户看到了什么/没有看到什么、以及原因属于哪一类边界（隐私、合规、质量、或安全）”。

#### F. 渠道契约与 WebApp 交互面

##### 渠道适配与降级合同

渠道与入口保持接口优先（interface-first），试点入口在 Telegram。后续对接其他端时，优先复用已完成的机器人能力与统一 API 合同：对方端支持的能力直接开放对应 API；不支持的能力明确标记为该端不可用（feature unavailable），并给出回执而不是静默失败。若是自研 App/Web 端，则可在统一 API 之上自研对应页面能力（包括 canvas 交互）。能力差异通过降级（degradations）与交接链接（handoff links）表达，不形成分裂真相。配置应在运营侧一次编写并跨入口共享，同时每个入口都应降级到其可靠支持的能力。

##### 统一Web UI承载策略

面向用户与管理员的 web UI 设计为共享的 HTML5 承载面：在 Telegram 中以 Web App 形式嵌入以承载默认终端用户体验，并固定为 mobile-first 竖屏尺寸。Admin UI 强制采用手机尺寸设计；桌面浏览器访问时也必须维持同一固定宽度/高度与竖屏布局，不提供横屏专用体验。该 UI 基座需同时服务管理员入口与未来用户自研入口，避免维护两套分裂前端。chat 用于指引与回执，Web UI 用于受控动作。

##### Telegram Chat+WebApp契约

Telegram 交互应被视为 chat + Web App 的契约，而不是 chat-only 的 UI spec。Telegram Web App 是导航与用户可控动作的默认承载面（历史视图、账户设置、登记/记录更新、结构化审核屏）。chat 仍作为 AI 层：解释用户正在做什么；在需要时请求明确确认；收集不需要完整页面的输入；并为可审计结果生成带 ID 与时间戳的回执。

##### Telegram原生能力拼接

使用 Telegram 原生能力把两者“缝合”起来：inline buttons 与 deep links 可以打开 Web App 到特定路由（routes）；`CallbackQuery` 可捕获意图并驱动状态迁移；文档流程（upload/preview/generated）可通过 `sendDocument` / `sendPhoto` 并在 chat 中生成回执。允许列表必须显式（`InlineKeyboardMarkup`, `CallbackQuery`, `ReplyKeyboardMarkup`, `ForceReply`, `editMessage*`, deep-link start parameters, `sendDocument`, `sendPhoto`），并将 Web App 视为主要 UI，而非可选增强。

##### 试点非目标与替代路径

试点的非目标（Non-goals）包括：在阶段一至二不接入真实支付接口，以及依赖不受支持的 Telegram 客户端特性。当 Telegram Web App 能力缺失或不可靠时，系统应降级为多轮 chat 表单与基于消息的导航。较大的预览可作为文档发送，并附带说明文字（caption）与快捷操作。

#### G. 数据治理、合规与硬边界

##### 数据留存与用户权利

数据留存与画像（profiling）仅允许在明确同意与合规前提下进行。系统保留用于回放、评估与最小画像所必需的对话与材料片段（偏好、阶段、意图变化、任务完成率）。画像仅用于提升回答相关性与指引质量；频率受限；理由可解释；范围可见；用户可查看/导出/更正/删除数据并撤回同意。

##### 合规行为与外部依赖降级

合规与信任必须落实为产品行为。数据收集应尽量最小化，并披露其使用方式，包括留存周期与使用范围。系统记录“为什么建议这一步 / 为什么打开这个视图”，关键节点产出可审计回执。敏感或不合规内容需要最小必要检测，并执行掩码或阻断与告警。外部依赖需做健康监测，并具备有计划的降级路径。

##### 法务与流程硬边界

存在硬边界：系统不提供法律建议，不做超出公开政策的承诺，不绕过官方流程或资格审查。

#### H. 后台治理、非承诺与成功标准

##### 后台治理与文档先行

管理后台（Admin Console）是远端部署的运营者/开发者控制台：必须登录并执行权限治理与审计；默认对普通用户不可见（未赋权账号无法看到后台入口，也不得枚举后台资源）；后台访问必须启用设备绑定。终端用户永不直接访问管理后台。文档治理必须显式区分两套资产：开发文档（开发线）与运营文档（运营线）；并且显式区分两套 prompt 资产：后台运维 prompt（operator-facing）与前台用户流程 prompt（consumer-facing）。前后端与流程变更在对齐实现前，必须先更新对应线的规范与使用文档，禁止跨线混写导致语义漂移。Admin 控制台必须提供两类可视面板：1）系统运行性能面板（可用性/时延/成功率/会话隔离）；2）章鱼OS文档推进面板（母里程碑/子里程碑状态、当前锚点与差异）。两类面板都要同时支持 AI 脚本读取与人类手机查看。

##### 非承诺边界

非承诺（Non-commitments）必须明确：管理后台不被视为终端用户产品；不使用复杂术语来甩锅决策；排除不受控的推送；禁止隐藏画像与未授权数据共享；功能堆叠不能替代问题解决；导航不得在无必要情况下变得不透明或多层化。

##### 成功标准三维度

成功标准是方向性的。对用户：更快获得可执行答案、更高的一次接触解决率、合理的人类交接率、并持续提升满意度。对运营：跨渠道、知识、工作流（workflows）、工具（tools）、变量（variables）的成熟可观测性（健康、告警、回放、归因），以及更快且更正确的部署。对业务：在合规前提下通过最小画像提升转化与留存；降低新场景试验成本；更快发布；并控制风险。

##### 全局角色拓扑与责任隔离

本项目采用四类 AI 角色拓扑并执行硬隔离。开发协作角色负责开发线文档与代码推进；运营中枢角色负责后台调度与运营治理；用户体验角色负责客户对话与页面引导；运行判定角色负责策略路由、降级与门禁裁决。角色边界通过统一命令协议与事件契约连接，但禁止跨域直接复用 prompt 语义。后台运维 prompt 与前台用户流程 prompt 必须保持资产隔离，任何跨域桥接都必须经由可审计的 typed contract。

角色主次关系补充：你（唯一人类）是最高指挥官（`human_commander_root`），对策略、放行、越权例外和最终责任拥有最高裁决权。Codex CLI 在本项目中定位为“开发+运维一体协作体”（`codex_devops_copilot`），负责执行已裁决任务与回报审计证据，但不拥有最终政策裁决权。

记忆与知识边界补充：章鱼OS运行时记忆在本项目中等价于代码库运维 AI 记忆（`project_devops_memory`）；`OctuposOS_Runtime_Backend` 下的文档产物可被视为该运维 AI 的项目技能知识面（`project_coupled_skill_knowledge`）。同时，章鱼OS技能本体必须保持通用规则/模板/工具，不得携带任何具体项目运行时正文内容。

##### 全局放行门禁与进入条件

全局放行门禁采用“先语义完整、再结构完整、再执行放行”的三层 gate。第一层要求 L0-L3 的必填语义主题完整且锚点可追溯；第二层要求 L1 合同状态达到可放行条件（至少达到 draft，关键合同达到 locked）；第三层要求链路门禁通过（可用性、时延、成功率、会话隔离、审计追踪）。任一 gate 未通过时，系统必须返回待补全状态与缺口清单，禁止直接进入后续开发落盘或运营放量。

代码与文档采用“混合落盘 + 晋升门禁”策略：允许在 `draft_shadow` 状态把文档与草稿代码同步落盘，以加速迭代和映射收敛；但只有当当前 topic 完成 `L3-L13` 纵向闭环并通过 full gate 后，才可晋升为 `slice_ready` 进入切片测试；只有当母里程碑全部 topic 完成并通过发布门禁后，才可晋升为 `release_ready` 进入上线与运营放量。状态晋升必须由机器判定证据触发，禁止仅凭叙事文本手工晋升。发布一致性硬约束：项目改动默认视为“代码改动 + 运行文档改动 + 代码映射改动”的同一变更事件，三者缺一不可。

#### I. 范围声明

##### 跨层义务映射

L0 对下游层的义务映射固定如下：L1 负责合同域拆分与约束落位；L2 负责主题扩展叙事与 L3 下钻映射；L3 负责线性执行语义合同（流程、裁决、回退、产出物）。从 L4 起允许进入更细开发落地，但只能消费上游已显式声明的语义与锚点，不允许跳层新增核心语义。每一层必须声明“来源层输入、输出给下一层的资产、为何必须这样交接”，确保链路可追溯、可审计、可回放。

##### 当前目标基线与冲突裁决规则

范围声明：本文档是以上述时间戳为准的当前“current goal”基线锚点。若它与更旧、更宽泛的愿景陈述冲突，则以本版本为准。能力扩展应通过后续 updates（使用更新的时间戳）表达，而不是静默重写历史。

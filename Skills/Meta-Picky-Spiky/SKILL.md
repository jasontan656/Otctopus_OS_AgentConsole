# Meta-Picky-Spiky

## Overview

这是一个只用于手工召唤的 meta 技能。

它不负责写文档，不负责修文档，也不负责生成执行步骤。
它只负责一件事：

- 通读目标文档，或者目标路径下的一组文档
- 挑出所有会导致后续理解、规划、实现或验收跑偏的**未裁决语义**

默认立场是：

- 不替作者脑补
- 不帮作者圆场
- 不因为“看起来大概能懂”就放过
- 只要某个表述仍然存在多种合理解释，并且这些解释会影响后续行为，就视为问题

## Review Lens

本技能关注的不是文笔好不好，而是语义有没有被裁决干净。

所谓“未裁决语义”，包括但不限于：

- 目标未裁决：到底要做到什么、做到什么才算完成，没有被锁死
- 路径未裁决：系统必须怎么实现、哪些链路必须经过、哪些 shortcut 禁止，没有被说清
- 边界未裁决：谁负责、谁不负责、哪些目录/系统/阶段属于范围内，没有被切清
- 阶段未裁决：当前阶段要产出什么、下一阶段依据什么进入、阶段之间是否允许混读，没有被裁定
- 验收未裁决：测试、真实 witness、交付完成条件、blocker 条件，没有被裁定
- 术语未裁决：同一个词在文档里承担了多个意思，或者不同词指向同一对象
- 失败语义未裁决：失败时如何暴露、是否允许降级、什么叫“阻塞”，没有被锁定
- 输入输出未裁决：谁提供输入、谁消费输出、文档与代码/计划/验收之间怎么映射，没有被明确
- 优先级未裁决：多项要求同时存在时，以什么为先，没有被写明
- 回退与替代未裁决：默认路径失败后是否允许切换替代路径，没有被界定

## Document Set Reading

当用户给的是一个路径而不是单文件时，本技能把这些文档视为一个整体系统说明来审。

此时不只挑单文档内部的问题，还要挑：

- 跨文档冲突
- 跨文档重复但不一致
- 某个关键裁决只在局部出现，其他文档仍保持开放语义
- 上游文档说得很硬，下游文档又把它说松了
- 阶段之间互相污染

如果多个文档共同描述一个系统，那么“跨文件仍未裁决”也属于违规。

## Output Contract

输出以 findings 为主，不做润色报告。

每条 finding 至少要说清：

- 在哪里
- 哪一句或哪一类表述有问题
- 为什么它仍然是未裁决语义
- 它会把后续什么行为带偏
- 当前真正缺的裁决是什么

如果问题很多，优先挑这些：

- 会改变实现路径的问题
- 会改变阶段行为的问题
- 会改变完成定义的问题
- 会让模型错误聚焦的问题

## Guardrails

- 不要把“作者大概是这个意思”当成已经裁决
- 不要因为表述高级、抽象、流畅，就忽略未裁决本质
- 不要把拼写、措辞、排版问题当成主要输出
- 不要直接进入修复方案长文
- 不要输出执行步骤、命令步骤、施工攻略
- 不要降低标准；挑得越细、越准，这个技能越有价值

## Success Signal

本技能的成功，不是“文档看起来还行”。

而是：

- 能把真正未裁决的地方揪出来
- 能指出这些地方为什么会污染后续模型心智
- 能让作者明确知道还缺哪一个裁决

# 使用合同

## 唯一目标
- 让模型先绑定长期任务，再把任务快照写入受管目录，并在当前与后续相关回合持续服从这份快照。

## 模型读取后必须形成的结论
- 本技能的第一动作不是直接执行实现，而是先判定 `new_task` 或 `continue_task`。
- 任务一旦绑定，就必须把有效语义写入 `/home/jasontan656/AI_Projects/Codex_Skills_Result/temp-plan`。
- 任务日志由 AI 维护，用户只需要自然语言说明是否新建任务、任务叫什么，以及后续纠偏内容。
- 记录内容以最小语义为准，更新当前真相优先于保留陈旧措辞。

## 模型接下来应怎么做
- 若用户明确说新任务：
  - 归一化任务名。
  - 创建新任务目录。
  - 写入 `ACTIVE_TASK.md`、`TASK.md`、`TURN_LOG.md`。
- 若用户明确说继续任务：
  - 找到对应任务目录。
  - 更新 `ACTIVE_TASK.md`。
  - 刷新 `TASK.md` 与 `TURN_LOG.md`。
- 若用户没有明确说新建或继续，但当前上下文只有一个高确定性任务：
  - 可以直接推断并继续。
- 若任务身份仍然模糊，且继续执行会污染错误目录：
  - 先补问“是新任务还是继续已有任务，以及任务名”。
- 若任务是创建、改造或安装新的开发技能：
  - 用本技能固化任务背景后，再转入 `$SkillsManager-Creation-Template`。

## 必记字段
- `collaboration_mode`：协同方式。
- `work_preference`：工作偏好。
- `task_goal`：任务目标。
- `common_rules`：共通规则。
- `user_taste`：用户喜好。
- `task_mindset`：当前任务心智模型。
- `extra_axes`：额外结构化方向。

## 推荐补充字段
- `scope_boundary`：做什么，不做什么。
- `decision_style`：如何裁决争议。
- `deliverable_shape`：交付偏好。
- `risk_guardrails`：明显不能踩的坑。
- `status`：`active` / `paused` / `done` / `archived`。

## 写法合同
- 使用短词组、短句、片段。
- 允许英文，不要求完整句。
- 不写长段解释，不写故事化总结。
- 同一信息在 `TASK.md` 中只保留一份当前版本。
- 历史变化写到 `TURN_LOG.md`，不是把 `TASK.md` 变成长日志。

## OK 定义
- 用户一提到任务名，模型能快速对齐该任务的协同方式、偏好、目标和边界。
- 日志目录稳定存在，一次任务一个目录。
- `TASK.md` 体现的是当前有效真相，`TURN_LOG.md` 体现的是增量变化。
- 任务目录一旦在当前上下文绑定，后续相关回合默认持续生效并必须遵守。

## Not OK
- 触发技能后不建任务目录。
- 只写任务目标，不写协同方式和偏好。
- 把长篇说明塞进任务日志。
- 用户纠正后仍保留旧偏好为主版本。
- 继续已有任务时另起一个重复目录。

## 边界提醒
- 本技能只负责任务级长期对齐，不替代具体 domain skill。
- 本技能不自带 CLI；markdown 日志就是主运行载体。
- 进入具体 repo 工作后，仍需服从 workspace 与 repo 的 `AGENTS.md` 合同。

# 使用合同

## 唯一目标
- 让模型在进入具体任务前，快速掌握用户当前的长期开发策略、技能化优先级和验收边界。

## 模型读取后应形成的结论
- 当前重点不是边开发边临时补公共定义，而是先把公共定义收敛为技能。
- 技能是公共框架、规范、复用资产与示例材料的首选承载方式。
- 后续若出现新的开发主题，应优先判断是否值得沉淀为独立技能。
- 一旦决定创建技能，具体结构必须遵循 `$skill-creation-template`。

## 模型接下来应怎么做
- 若用户要创建或整理新的开发技能：
  - 先用本技能理解背景。
  - 再进入 `$skill-creation-template` 做具体骨架生成与治理。
- 若用户要做具体开发任务：
  - 先判断这件事需要的是“新技能沉淀”还是“直接实现”。
  - 若缺的是公共定义，优先补技能，不要把公共内容直接散写进 repo。
- 若用户已经有相应技能：
  - 优先复用现有技能中的规范、示例和结构，不重新发明一套。

## OK 定义
- 模型能准确复述这三件事：
  - 用户现在在做什么：逐步把开发相关主题沉淀成技能。
  - 用户为什么这样做：把公共定义集中管理，减少散落与重复解释。
  - 什么算完成：新技能能承载公共定义并遵循 `$skill-creation-template`，后续开发可直接在技能规范下展开。
- 用户无需在每次相关回合里重复补充这段长期背景。
- 模型不会把本技能误用成具体技术域技能或 repo 运行合同。

## Not OK
- 把公共定义继续散落在项目实现文件、零碎文档或临时回复里。
- 把本技能当成任何具体技术栈的完整实施规范。
- 创建新技能时跳过 `$skill-creation-template`，转而使用随意结构。
- 在没有判断复用价值前，就直接把长效规范写进一次性任务上下文。

## 边界提醒
- 本技能只负责背景注入，不替代具体技能的 domain 内容。
- 本技能不包含 CLI，不承担 lint、安装、生成或 repo 操作。
- 一旦进入具体 repo 工作，仍需服从该 workspace 与 repo 的 `AGENTS.md` 合同。

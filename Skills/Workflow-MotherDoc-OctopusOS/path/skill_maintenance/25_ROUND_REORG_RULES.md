# 回合级结构调整与内容迁移规则

## 每一回合的最小动作顺序
1. 识别用户输入中的双重信号：当前要开发/调整什么，以及这说明本技能自身缺了什么稳定承载位。
2. 先判断问题属于“改正文”还是“改结构”；若只是表述不清，优先改现有正文，不急于新增文件。
3. 需要改结构时，优先做最小有效动作：重写当前文档、关键词优先替换、拆分、合并、移动、重命名。
4. 一旦发生结构迁移，同步搬运既有正文，更新入口链、索引、引用与测试，不保留长期并行真源。
5. 回合结束前再次判断：当前骨架是否比上一轮更贴近真实使用过程，而不是仅仅多了一层说明。

## 允许的结构动作
- 新增承载位：当某一类主题持续出现，且现有文档无法稳定容纳时新增。
- 拆分：当一个文档混合了不同职责、不同读者或不同维护节奏时拆分。
- 合并：当多个文档长期稀薄、重复或始终成对阅读时合并。
- 迁移：当正文明显处在错误层级、错误目录或错误入口下时迁移。
- 重命名：当现有名字无法稳定表达职责边界时重命名，并同步更新引用。

## 正文迁移硬规则
- 同一段治理语义只能保留一个主真源。
- 迁移后必须删除或重写旧位置，不能留下内容并行的“新旧双份正文”。
- 若旧位置仍需保留发现性提示，应只保留短提示，不重复承载完整规则。
- 任何迁移都要优先保持“首次阅读路径更清晰”，而不是为了局部美观制造更深层级。

## 结构决策触发条件
- 流程类信息反复出现但总散落在多个文档中。
- 某个文档已经同时承担规则、示例、边界和执行说明，导致职责失焦。
- 某一主题已经影响到后续多个回合，却还只能塞在临时段落里。
- 同一主题既影响 `mother_doc` 写回行为，又影响本技能自己的维护方式，需要拆出独立治理承载位。

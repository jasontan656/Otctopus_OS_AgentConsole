# 当前产品意图

## 当前产品身份
- 产品品牌：`Octopus OS`
- 当前工程仓名称：`octopus-os-agent-console`
- 当前对外定位：`自然语言驱动的多 Agent 控制台`
- 当前起点角色：`一人公司运营者`

## 当前核心判断
- 当前产品的主内核不是单个工具，而是“技能体系 + 技能编排 + agent 控制协议 + 可安装/可清理运行面”。
- 这个产品不是为了展示单点 AI 能力，而是为了给一人公司运营者提供从 0 到 1 搭建完整协同助手能力的路径。
- 目标未来形态是：协同助手可以覆盖开发、部署、运维、市场、多 AGENT 协作、自动化、技能流程编排与自然语言开发。
- 这个产品仓要同时承担：
  - 对外产品形态
  - 对内 skill 迭代源
- skill 仍然必须先在本仓演进，再推送到 `~/.codex/skills`。
- 产品门面、产品文档、产品工具不能进入 codex 安装目录污染技能执行面。
- 长期目标是把 AI 所需流程全部内建并定制到产品自身，而不是依赖外部技能安装。
- 这样做的根本目的，是从流程层面控制 AI 在每个细节上的行为边界，降低外部依赖与注入攻击面，让能力完全安全可控。

## 当前语言边界
- 对外产品 `README.md` 与 `docs/`：英文
- 面向终端用户的 wizard / TUI：中英双语
- 内部 skill 内核、治理合同、内部开发记录：允许中文为主
- GitHub 上的产品迭代 commit subject 与迭代日志：优先英文

## 当前产品施工目标
- 把本地仓从旧的 `Codex_Skills_Mirror` 语义，收敛成真正的产品形态基础。
- 在不破坏内部 skill push 能力的前提下，形成清晰的产品门面、安装模型、清理模型与语言规范。
- 让对外文档能讲清楚“一人运营者如何从零搭建自己的完整 AI 协同体系”。
- 逐步把当前产品的能力叙事从“技能集合”升级成“可安装、可理解、可控的自带协同助手系统”。
- 为未来的 skill composition、agent orchestration 与真实项目开发做稳定过渡。

## 当前阶段性策略
- 在完整产品形态切换完成前，设计变更历史先写在本地 markdown。
- 这个阶段的本地设计日志，是为了让 AI 持续读取“为什么现在是这样”的上下文。
- 等产品形态足够稳定，再逐步把主迭代历史切回 GitHub。
- 当前版本是极不稳定的高速迭代构建，推送频率可能达到每 10 到 15 分钟一次。
- 因而当前阶段并不适合让外部读者把 commit 流水当成正式文档；正式学习入口应优先依赖产品文档与结构化说明。

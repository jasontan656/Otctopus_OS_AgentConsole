# Topic Anchor Graph Contract

## 1. 核心定义
- `topic tree`：父子层级组成的主导航路径，负责把读者从大域收敛到小域。
- `anchor graph`：在主导航树之外，为 sibling、依赖、前置、跨层补充的关系边。
- `node`：一个文档单元。节点可以是导航节点，也可以是原子节点。
- `topic atom`：最小可读且语义不再漂移的节点；继续拆分已无显著收益。

结论：
- 这不是普通目录树。
- 也不是没有主脊梁的任意图。
- 正确形态是：`tree as spine + anchors as edges`。

## 2. 单 topic 约束
每个节点必须只回答一个稳定问题，或只承载一个稳定职责。

以下任一命中，都视为 topic 不纯，必须继续拆：
- 标题里自然需要出现 `和`、`以及`、`兼` 才说得通。
- 同一节点同时服务不同消费者，且消费者真正关心的结论不同。
- 同一节点同时包含定义、决策、执行步骤、验收而彼此都已足够复杂。
- 读者必须在同一节点里频繁跳过无关段落才能得到答案。
- 节点内容一旦删除其中一半，另一半标题就该改名。

## 3. 节点角色
节点角色不绑定层深，只绑定语义职责：

### 3.1 导航节点
- 负责定义当前域的边界。
- 负责列 child nodes 与阅读顺序。
- 只保留最小摘要，不吞下游细节。

### 3.2 原子节点
- 负责承载一个清晰 topic 的完整结论。
- 可以包含局部规则、输入输出、例外、边界。
- 不再继续当作索引页使用。

## 4. 锚点合同
anchor 用来表达“有关系，但不应该为了这层关系合并文档”。

推荐 anchor 类型：
- `parent`：主路径父节点。
- `child`：主路径子节点。
- `related`：同层相关，但无前后依赖。
- `prerequisite`：前置阅读。
- `depends_on`：该节点成立依赖另一个节点。
- `refines`：本节点是上游节点的更细分版本。
- `cross_layer`：跨层必要关联。

anchor 必须满足：
- 可显式列出目标节点或目标路径。
- 能说明关系类型，不只给裸链接。
- 不代替正文；anchor 只负责导向，不负责重复抄写对方内容。

## 5. 写作协议
创建或重构文档时，按这个顺序执行：
1. 写出当前最大域的主题句。
2. 把内容切成若干 `topic candidates`。
3. 对每个 candidate 做 single-topic 检查。
4. 为通过检查的 candidate 分配节点；没通过的继续拆。
5. 建主路径：谁是父，谁是子，谁只是 related。
6. 只把路由与摘要留在上游，细节全部下沉。
7. 为任何横向、依赖、跨层关系补 anchor。

## 6. 编辑协议
当任务是编辑已有文档时：
- 优先判断“这段内容是否该留在当前节点”。
- 若不该留，优先迁移/拆分/重组，不要继续在原位追加。
- 若多个节点都在重复同一语义，保留最权威节点，其余改成 anchor。
- 若新增内容会污染当前节点 topic，必须新建节点而不是硬插小节。

## 7. 上下游分工
- 上游节点负责：
  - 域边界
  - 路由
  - 子节点索引
  - 最小摘要
- 下游节点负责：
  - 具体规则
  - 细节
  - 局部例外
  - 原子化结论

上游节点不得重新解释下游全文；否则会重新长成大杂烩文档。

## 8. 反模式
- 用一个大文档承载多个平级 topic，只靠标题编号假装结构化。
- 把 anchor 藏在自然语言里，不显式标注关系。
- 上游节点复制下游大量正文，导致“双源真相”。
- 为了减少文件数，故意把多个应拆 topic 合并。
- 把层深当成固定模板，导致结构不能按语义自然扩展。

## 9. 嵌入其他技能或文档流时的最小要求
当本方法论嵌入其他技能生成或编辑流程时，至少执行：
- 先列出 topic candidates，再开始写文件。
- 每个输出文件只绑定一个稳定 topic statement。
- 每次新增文件时，同时声明其 parent path 与 anchors。
- 若内容仍然过大，继续拆，而不是先写一整块再寄希望于后续整理。
